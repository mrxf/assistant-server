# 认证体系设计方案

> 文档版本：v1.0

## 一、背景与目标

assistant-server 是一个基于 NestJS 的 AI 助手后端服务，当前所有 API 接口完全暴露，无任何认证与鉴权机制。在上线部署前，需要引入认证体系以确保：

1. **接口安全**：未经授权的请求无法访问任何业务接口
2. **数据隔离**：用户只能操作自己的数据，无法越权访问他人资源
3. **管理分级**：区分管理员与普通用户，敏感操作（调试、用户管理）仅管理员可用
4. **受控注册**：通过邀请码机制控制用户增长，防止滥用

## 二、技术选型

| 决策项 | 选型 | 理由 |
|--------|------|------|
| 认证框架 | NestJS Passport + JWT | NestJS 官方推荐，与项目技术栈完美契合 |
| 密码存储 | bcrypt | 业界标准，自带 salt，抗彩虹表攻击 |
| Token 策略 | Access Token + Refresh Token | 兼顾安全性与用户体验 |
| 注册控制 | 邀请码制 | 可控增长，适合小规模上线 |

### 新增依赖

| 包名 | 用途 |
|------|------|
| `@nestjs/passport` | NestJS 的 Passport 集成模块 |
| `@nestjs/jwt` | JWT 签发与验证 |
| `passport` | Passport 核心库 |
| `passport-jwt` | JWT 认证策略 |
| `bcrypt` | 密码哈希 |

## 三、核心设计决策

### 3.1 用户与玩家关系：合并模型

**决策**：直接在现有 `Player` 表上扩展认证字段，Player 即 User。

**理由**：
- 项目为单角色场景（每人一个助手），无需独立 User 表
- 减少表连接，查询更简单
- `username` 直接作为 `Player.id`，语义清晰

### 3.2 登录方式：用户名 + 密码

**决策**：使用 username + password 方式登录。

**理由**：
- 不依赖邮箱验证或短信服务，零外部成本
- 用户名即 Player ID，一致性好
- 适合工具型产品的简洁登录体验

### 3.3 Token 策略：双 Token 机制

| Token 类型 | 有效期 | 存储位置 | 用途 |
|-----------|--------|---------|------|
| Access Token | 15 分钟 | 客户端内存/LocalStorage | 接口认证，短周期高频使用 |
| Refresh Token | 7 天 | 客户端持久存储 + 服务端数据库 | 无感续签 Access Token |

**流程**：
1. 用户登录 → 服务端签发 Access Token + Refresh Token
2. 客户端请求带 `Authorization: Bearer <access_token>`
3. Access Token 过期 → 客户端用 Refresh Token 调用 `/auth/refresh` 获取新 Access Token
4. Refresh Token 过期 → 需要重新登录

### 3.4 注册策略：邀请码制

**决策**：多次使用邀请码，每个邀请码有使用次数上限。

**规则**：
- 管理员通过 `/auth/invite` 生成邀请码
- 每个邀请码可设定最大使用次数（默认 5 次）
- 注册时需提供有效的邀请码
- 邀请码记录创建者和使用次数，便于追溯

### 3.5 角色权限：简单 Admin 标记

**决策**：在 Player 上增加 `isAdmin` 字段，二元区分管理员与普通用户。

| 角色 | 权限范围 |
|------|---------|
| 普通用户 | 仅可操作自己的 Player 数据（聊天、事件、消息、NPC 状态等） |
| 管理员 | 全部普通权限 + 查看/删除任意用户 + Debug 接口 + 管理邀请码 + 访问 Swagger 文档 |

## 四、数据模型变更

### 4.1 Player 表改造

在现有字段基础上新增认证字段：

```prisma
model Player {
  id           String   @id          // = username，用作登录标识
  passwordHash String                // bcrypt 哈希后的密码
  isAdmin      Boolean  @default(false)
  nickname     String   @default("")
  birthday     String?
  customFields String   @default("{}")
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}
```

### 4.2 新增 InviteCode 表

```prisma
model InviteCode {
  id        String   @id @default(uuid())
  code      String   @unique        // 邀请码字符串
  maxUses   Int      @default(5)    // 最大使用次数
  usedCount Int      @default(0)    // 已使用次数
  createdBy String                  // 创建者 Player ID
  createdAt DateTime @default(now())
}
```

### 4.3 新增 RefreshToken 表

```prisma
model RefreshToken {
  id        String   @id @default(uuid())
  token     String   @unique        // Refresh Token 值
  playerId  String                  // 关联的 Player ID
  expiresAt DateTime               // 过期时间
  createdAt DateTime @default(now())
}
```

## 五、API 端点设计

### 5.1 新增认证端点

| 方法 | 路径 | 认证要求 | 说明 |
|------|------|---------|------|
| `POST` | `/auth/register` | 无（需邀请码） | 用户注册 |
| `POST` | `/auth/login` | 无 | 用户登录 |
| `POST` | `/auth/refresh` | 无（需 Refresh Token） | 刷新 Access Token |
| `POST` | `/auth/invite` | 管理员 | 生成邀请码 |
| `GET` | `/auth/invites` | 管理员 | 查看邀请码列表及使用情况 |

### 5.2 注册接口

```
POST /auth/register
Body: { username: string, password: string, inviteCode: string, nickname?: string }
Response: { id, nickname, accessToken, refreshToken }
```

### 5.3 登录接口

```
POST /auth/login
Body: { username: string, password: string }
Response: { accessToken, refreshToken, player: { id, nickname, isAdmin } }
```

### 5.4 刷新接口

```
POST /auth/refresh
Body: { refreshToken: string }
Response: { accessToken, refreshToken }
```

## 六、路由保护规则

### 6.1 公开路由（无需认证）

| 路径 | 说明 |
|------|------|
| `POST /auth/register` | 注册（需邀请码验证） |
| `POST /auth/login` | 登录 |
| `POST /auth/refresh` | Token 刷新 |
| 微信 Webhook 路由 | 由 `@innerlife/channel-nestjs` 内部管理，有微信签名验证 |

### 6.2 认证路由（需 JWT + 归属校验）

所有 `/player/:playerId/*` 路由，需同时满足：
- 携带有效的 Access Token
- Token 中的 `playerId` 与路由参数 `:playerId` 一致

| 路径 | 说明 |
|------|------|
| `GET/PUT /player/:playerId/profile` | 个人资料 |
| `POST /player/:playerId/chat/send` | 发送聊天消息（SSE） |
| `GET /player/:playerId/chat/history` | 聊天历史 |
| `POST /player/:playerId/wechat/bind` | 微信绑定（SSE） |
| `DELETE /player/:playerId/wechat/:peerId` | 解绑微信 |
| `POST /player/:playerId/events` | 事件上报 |
| `GET /player/:playerId/messages` | 消息列表 |
| `GET /player/:playerId/messages/pending` | 待读消息 |
| `POST /player/:playerId/messages/:id/read` | 标记已读 |
| `GET /player/:playerId/npc/status` | NPC 状态 |

### 6.3 管理员路由（需 JWT + isAdmin）

| 路径 | 说明 |
|------|------|
| `GET /players` | 查看所有玩家列表 |
| `DELETE /players/:playerId` | 删除任意玩家 |
| `POST /messages/unread-counts` | 批量查询未读数 |
| `POST /auth/invite` | 生成邀请码 |
| `GET /auth/invites` | 查看邀请码列表 |
| `GET/POST /player/:playerId/debug/*` | 调试端点 |
| `POST /debug/reset-all` | 全局重置 |
| `POST /debug/test-llm` | LLM 测试 |
| `GET /api/docs` | Swagger 文档 |

## 九、首位管理员创建

系统上线时数据库为空，需要一种方式创建首位管理员。方案：

1. 提供一个 CLI 脚本或 seed 命令：`pnpm prisma:seed`
2. 脚本读取环境变量 `ADMIN_USERNAME` 和 `ADMIN_PASSWORD`
3. 创建 `isAdmin: true` 的 Player 记录
4. 由该管理员生成邀请码，分发给其他用户注册

## 十、安全注意事项

1. **JWT Secret**：生产环境必须使用强随机密钥，不得硬编码
2. **密码强度**：建议注册时校验密码最小长度（≥ 8 位）
3. **Refresh Token 轮换**：每次使用 Refresh Token 时签发新的，旧的立即失效
4. **CORS 收紧**：生产环境应配置具体的 `origin` 白名单，不使用 `*`
5. **Rate Limiting**：登录接口建议添加频率限制，防止暴力破解
6. **HTTPS**：生产环境必须通过 HTTPS 传输，防止 Token 被截获
